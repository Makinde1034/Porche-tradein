Usage

Integrate the version you need (dev for Preview or dist for Production) into your page and instantiate by calling the constructor: 

var GDL = new GlobalDataLayer(mySettings);

On the most porsche.com pages you can use the already available global instance.

window.GDL.push(myData);

Versions

dev for Preview and Test

The dev version can validate your data if you also transfer the eventMatrix setting.

dist for Production

Optimized for production with no complex validation available.

Constructor Arguments

 
var GDL = new GlobalDataLayer({eventMatrix: myValidationData});

GDL accepts a settings object to modify certain default configuration. These are the properties that can be passed:

Argument Param Description
settings eventMatrix Object It is the object which will be use to perform the validations. This object is generated through the command generate-event-matrix. The validation is only available in the dev version.
removeUnacceptableValues Object Contain the custom configuration to remove automatically certain values of the objects passed.
allowUnknownPropsOnPush boolean Allow to push properties that are not exist in the given eventMatrix. This can be overridden in the options of every push.

For more information about the settings object please read the Docs generated for the project using the JSDoc in the folder ./docs/.

Methods

push

Push data to the pagData global array by validating it against the matrix located on src/layers/[YOUR_PROJECT]/data/events.json.

Param Description
event String The event you want to push.
eventData Object Required properties for the event. Please note that these data might be merged with the properties transferred already via the setDefaultProperties method.
options Object Set of options
options.turnOffValidation boolean Turn off the validation of the given eventData against the given eventMatrix.
options.allowUnknownProps boolean Allow to push properties that are not exist in the given eventMatrix.

setDefaultProperties

Set default properties to one or multiple events to avoid repeating.

Param Description
event String The event name to bind the properties to.
String[] Multiple event names to bind the properties to.
RegExp A Regular Expression to match event names to bind the properties to.
defaultProperties Object The default properties you want to set up.

Dynamic Values

You can also push dynamic values, a function that is always called at the moment some pushes an event. 

setDefaultProperties(
  'PAGPCOM_General_Pageload', 
  {
    visitor: {
      deviceBrowserHeight: () => window.innerHeight.toString(),
    },
  }
);

getPagData

Initialize the global array pagaData, if it is not already available, and return it.

registerExtension

Register an extension into the GDL instance.

Param Description
ext Object extends BaseExtension It is the extension class to register.
config Object Its a configuration object for the extension.

Extensions

An extension is a class to allow the user to run some logic across the GlobalDataLayer, providing a way to track certain things without the need to create the same logic in several places. \

Just imagine that you need to track the position of several DOM elements in the viewport. With an extension you would write the logic to be able to register the DOM elements and then you only need to call that extension through GDL.ext.MyExtensionName.register.

Creating an Extension

Create your extension class inside the directory src/extensions. Then extend your class with src/extensions/BaseExtension.ts.

export default class MyExtensionName extends BaseExtension {}

Inside the class you can create as many methods and logic as you want.

Loading an Extension

As the extensions are not loaded automatically in the GDL class, you will need to register your extension to make it available later. The place to do that is inside of your module for example. 

globalDataLayerInstance.registerExtension(
  new MyExtensionName(globalDataLayerInstance, { "mySetting": "withValue" })
);

Note that the second parameter could contain any additional setting or information to set up your extension.

Calling an Extension Method

After doing that you will be able to access to your extension through the ext property inside GDL. It would be like:

 GDL.ext.MyExtensionName.register();

In this example we are accessing to the register method we created inside our extension, but you can create the methods you want.

Available Extensions
  • DefaultWebProperties: To handle default properties, that could be used in every web application.
  • PageNameExtender: To handle the page name in single page applications or modules
  • ViewportTracker: To track DOM elements come into view

Working on The Library

Project structure
  • /data Contains the [YOUR_PROJECT].csv which is used to generate the EventMatrix.
  • /docs The output folder for the generated docs.
  • /dev The build folder for development.
  • /dist The build folder for production.
  • /src The code of the library.
    • /core The GlobalDataLayer core class.
    • /examples The example file to see the library working. You can use it as well to test the builds quickly.
    • /extensions Find BaseExtension and all global available extensions here.
    • /layers Contains all the projects and its respective generated EventMatrix.
    • /libraries Contains all the internal libraries used in the project.
  • /tests Contains all the unit tests for the library.
  • /webpack Contains the webpack config files.

In the root of the project you can find the config files for Prettier, Jest, Eslint and typescript.

events.json

How to Generate the Event Matrix

The event matrix is a JSON file that lives under /src/layers/[YOUR_PROJECT]/events.json. It lets the library know what the possible and required properties for an event are, in order to be able to validate the properties that were pushed to the global array via the push method.

Setup

To generate the matrix you need to update the /data/[YOUR_PROJECT].csv file of your project first. You MUST be careful with the name you give to the csv file because, it will generate the matrix inside of a folder with the same name. In the case of PAGPCOM, the csv has to be named like this 'PAGPCOM' (The naming conventions are simple: Only capital letters).

The json data will be part of the build output under /dist/events/. So you can use it in your project like this.

// dist/events/GlobalDataLayer.events.PAGPCOM.js
// dist/GlobalDataLayer.js
var GDL = new GlobalDataLayer({
  eventMatrix: GlobalDataLayerEventsPAGPCOM // insert your project key instead of PAGPCOM
});
The CSV File

The .csv file contains the worksheet Herleitung DL of the Excel file that comes from Smart Digital, which last updated version can be found under PCOM Tracking Events Overview.

To pass it to CSV just save it as "CSV UTF-8 (Comma delimited)" format with the O365 Excel app and set the delimiters as ';' (see this blog post for more information).

After you have updated the /data/[YOUR_PROJECT].csv, just run the command npm run generate-event-matrix and it will generate the new events.json and the typescript Interfaces (which currently are not in use) under src/layers/[your_CSV_file_name]/.

Validation via EventMatrix

The src/layers/[YOUR_PROJECT]/data/events.json allows the library to validate the events.

Find all available events in the events array (through that every event has an index):

{
  "events": [
    "PAGPCOM_FindADealer_Pageload",
    "PAGPCOM_FindADealerSearch_Click",
    "PAGPCOM_FindADealerLocateMe_Click",
    "PAGPCOM_FindADealerSearchResults_Load"
  ]
}

Find all properties named by there full key as an object:

{
  "visitor.deviceBrowserBreakpoint": {
    "t": "STRING",
      "e": {
        "0": 0,
        "1": 0
      },
    "r": 1
  }
}

All event properties have these three properties :

  • t: The type of the property (string, number, boolean, ...).
  • e: The events that have this property. The numbers correspond with the index of an event inside the events array (this concept keeps the file as light as possible). The value 0 is dummy value.
  • r: The mandatory option. 1 property is mandatory property and therefore, it always has to be passed for all the events. 0 not mandatory.

Pre Commit Hooks

There are two pre-commit hooks

  1. Checks if all the files have the proper format. So please set up your eslint and prettier properly.
  2. Runs all the tests inside /tests folder. Please make sure all the test succeed, otherwise you will not able to commit.

If you doubt which of these tests failed, you can run them separately (npm run lint or npm run test).

Naming Conventions

Branch

For creating branches and there naming conventions we created a Branching and Merge Concept in our confluence documentation section.

Commits

For the commits we have the following convention:

[CHANGELOG_PREFIX][TICKET_ID] [SHORT_SUMMARY_OF_THE_CHANGES_DONE]

Please keep the summary very short. You can also write a body message this needs some more explanation.

If a commit should be part of the automatic generated CHANGELOG.md you can also use the given prefixes.

⚠ Please note: Most of the automated changelog should come from the Merge Requests (which have a naming convention also). So please only add these prefixes if you have important log entries that are inside of a bigger change.

Merge Requests

For the naming conventions of merge requests we created a Branching and Merge Concept in our confluence documentation section.

All merge request are part of the automatic generated CHANGELOG.md. You can also use the given prefixes to define a category. Uncategorized merge requests are always features.

Merge Backs must always be done via a Merge Request and need to be prefixed to exclude them from the automatic changelog.

Auto Changelog Prefixes
Prefix Purpose Example
Merge request with no prefix are always categorized as features. DLPCOM-123 With crazyMethod we can now solve the balanced problem
MERGE BACK: or MB: A merge back merge request must be excluded from the changelog. _MB: master into feature/DLPCOM-123_add-crazy-method_
BREAKING CHANGE: Changes to the code that are important for others. BREAKING CHANGE: DLPCOM-123 crazyMethod has settings parameter now
BUGFIX: If you are fixing a bug please also mention the ticket then. BUGFIX: DLPCOM-123 crazyMethod fails when settings are undefined
CSV: Changes regarding a DataLayer Excel update CSV: PAGPCOM v123
TEST: If you have adapted or added an important jest test. TEST: Added test for crazyMethod
DOCS: If you have adapted or added an important documentation. DOCS: Added documentation for crazyMethod

Create a release

The process to create a new release is documented in detail on the Release Management confluence page.

Here is a short version:

  • Releases (or builds) are only allowed to be generated and pushed in the master. Therefore, you need to be a maintainer of the project.
  • Use the command of the release section like relese-fastlane.
  • If you have doubts, use release command and check the files before they are committed, tagged, pushed and released.
